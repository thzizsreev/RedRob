# Track A Architecture Migration: INSTRUCTOR-large Vectorization
### Redrob Hackathon — Candidate Retrieval System

---

## 1. Executive Summary

This document defines the architecture change for **Track A** — the dense semantic vector space used for Stage 1 candidate retrieval.

The original plan described a generic 768-d concatenated vector built from three 256-d subspaces. This document replaces that with a precise implementation using **`hkunlp/instructor-large`**, covering the model's internal architecture, why it fits our use case, how the 3-block orthogonal structure is constructed, and all optimizations needed to run within the hackathon's compute constraints.

---

## 2. What Changed and Why

### Original Plan (plan.md)
```
Three 256-d sub-vectors concatenated into one 768-d vector.
Sub-vectors generated by "an LLM offline."
Stored in FAISS HNSW, queried with Inner Product.
```

### Problems with Original Plan
| Problem | Impact |
|---|---|
| "LLM offline" was undefined — no specific model | Unimplementable as written |
| No mechanism for aspect isolation per block | Retrieval, Infra, Eval signal mixed into single embedding |
| Assumed generic encoder (sentence-transformers) | Cannot extract subspace-specific features from same text |
| 256-d per block was arbitrary | Information loss from unnecessary dimension restriction |

### New Architecture
```
career_history text (concatenated descriptions)
        ↓
INSTRUCTOR-large — 3 separate encode calls, 3 custom instructions
        ↓
retrieval_vec [768-d] + infra_vec [768-d] + eval_vec [768-d]
        ↓
concatenate → final_candidate_vec [2304-d]
        ↓
FAISS HNSW index — Inner Product similarity
```

## 3. INSTRUCTOR-large: Internal Architecture

### Backbone
INSTRUCTOR-large is built on **GTR-Large**, which is itself initialized from **T5-Large** (encoder-only).

```
T5-Large (770M params)
    → fine-tuned on information retrieval datasets → GTR-Large
        → fine-tuned on 330 task instructions with contrastive loss → INSTRUCTOR-large
```

### How a Single Encode Call Works

Given an instruction `I` and a passage `x`:

```
Input:  [I ⊕ x]  ← instruction and text are concatenated as token sequence
        ↓
T5 Encoder (transformer layers with self-attention)
        ↓
Last hidden representations → shape: [seq_len × 768]
        ↓
Mean pooling over tokens belonging to x ONLY (not instruction tokens)
        ↓
Output: fixed 768-d vector E(I, x)
```

**Critical detail:** Mean pooling is applied only to the tokens of the passage `x`, not the instruction tokens. The instruction influences the attention weights (what the model focuses on in `x`) but does not contribute to the pooled output vector.

This is the architectural mechanism that makes INSTRUCTOR different from simply prepending text to a standard encoder. The instruction tokens direct attention without polluting the output representation.

### Hard Negatives Checkpoint
The latest `hkunlp/instructor-large` checkpoint (updated 12/28) was retrained with **hard negatives** — negative examples that are semantically close but factually incorrect. This significantly improves the model's ability to discriminate between candidates with superficially similar but actually different experience.

For our use case: a Kafka/Spark engineer (Ira Vora) and a FAISS/retrieval engineer both have technical ML-adjacent vocabulary. The hard-negatives training makes the model push these apart in vector space, which is exactly the discrimination we need.

## 4. The 3-Block Orthogonal Architecture

### Why 3 Blocks, Not 1

A single encode call produces a vector representing the overall semantic content of the passage. Two candidates with very different technical profiles — one strong in retrieval but weak in evaluation, one the inverse — would land at similar distances from a single JD vector because their overall semantic similarity to the JD is comparable.

The 3-block architecture forces independent scoring on three axes:

```
score(query, candidate) = dot(q_ret, c_ret) + dot(q_inf, c_inf) + dot(q_eval, c_eval)
```

A candidate who is strong in retrieval but has zero evaluation experience scores high on block 1 and near-zero on block 3. That weakness is structurally preserved in the final score. A flat single vector would wash it out.

### Why the Math Supports This

Concatenation enforces structural orthogonality:

```
retrieval_vec_final  = [ret_768  |  zeros_768  |  zeros_768]
infra_vec_final      = [zeros_768 |  inf_768   |  zeros_768]
eval_vec_final       = [zeros_768 |  zeros_768 |  eval_768 ]

candidate_vec = retrieval_vec_final + infra_vec_final + eval_vec_final
              = [ret_768 | inf_768 | eval_768]
```

Dot product decomposes exactly:

```
dot([q_ret | q_inf | q_eval], [c_ret | c_inf | c_eval])
= dot(q_ret, c_ret) + dot(q_inf, c_inf) + dot(q_eval, c_eval)
```

Each block contributes independently. The orthogonality is structural (enforced by concatenation), not probabilistic.

**This is why Inner Product is required in FAISS, not Cosine Similarity.** Cosine normalizes the full 2304-d vector globally, destroying the magnitude differences between blocks. A candidate with near-zero evaluation experience gets normalized upward. Inner Product preserves the magnitude signal per block.

### Why 768-d Per Block (Not 256-d as Originally Planned)

Slicing a 768-d output to 256-d discards 512 dimensions of information per block. There is no principled reason to do this. The encoder's 768-d output space uses all dimensions. Artificially truncating it introduces information loss without any benefit.

The final vector is 2304-d. FAISS HNSW handles this efficiently — there is no FAISS limitation that requires 768-d total.

## 5. The Three Instructions

Instructions follow INSTRUCTOR's template:
```
Represent the [domain] [text_type] for [task_objective]:
```

### Block 1 — Retrieval & Ranking (dimensions 0–767)
```python
RETRIEVAL_INSTRUCTION = (
    "Represent the AI engineering career history for retrieving candidates "
    "with production experience in semantic search, embeddings-based retrieval, "
    "hybrid search systems, and ranking pipelines:"
)
```

**Targets:** Candidates who built real retrieval/search/recommendation systems, handled embedding drift, deployed vector databases, designed hybrid retrieval combining dense and sparse signals.

**Correctly scores near-zero:** Data engineers (Kafka, Spark, Airflow), frontend engineers, candidates whose only retrieval mention is "used Elasticsearch for logging."

### Block 2 — Infrastructure & Scale (dimensions 768–1535)
```python
INFRA_INSTRUCTION = (
    "Represent the AI engineering career history for retrieving candidates "
    "with production experience deploying and scaling ML systems, vector databases, "
    "inference optimization, and MLOps infrastructure:"
)
```

**Targets:** Candidates who deployed models to production, optimized latency/throughput, managed FAISS/Qdrant/Pinecone indexes at scale, worked with Kubernetes, Triton, ONNX, or similar infrastructure.

**Correctly scores near-zero:** Candidates whose "infrastructure" is only dev tooling (Docker for local dev, GitHub Actions), or pure data pipeline engineers.

### Block 3 — Evaluation & Rigor (dimensions 1536–2303)
```python
EVAL_INSTRUCTION = (
    "Represent the AI engineering career history for retrieving candidates "
    "who have designed evaluation frameworks for ranking systems, run A/B tests, "

)
```

**Targets:** Candidates who owned the measurement layer — built eval pipelines, ran controlled experiments, tracked retrieval quality regression, designed benchmark datasets.

**Correctly scores near-zero:** Candidates who used A/B testing for UI experiments, or who mention "metrics" only in the context of business KPIs.

## 6. Text Input Construction

### What Goes Into the Passage

Only `career_history[].description` fields, concatenated in reverse chronological order (most recent first). This gives the model the most relevant recent experience at the start of the passage, which is where transformer attention is strongest.

```python
def build_career_text(candidate: dict) -> str:
    # Sort by start_date descending (most recent first)
    sorted_history = sorted(
        candidate['career_history'],
        key=lambda x: x['start_date'],
        reverse=True
    )
    descriptions = [
        job['description']
        for job in sorted_history
        if job.get('description', '').strip()
    ]
    return " ".join(descriptions)
```

### What Is Explicitly Excluded

| Field | Reason for Exclusion |
|---|---|
| `skills[].name` | Self-reported, untrustworthy — Ira's Milvus/NLP inflation problem |
| `profile.summary` | Self-promotional, semantically redundant with career history |
| `profile.headline` | Too short for meaningful embedding |
| `education[].institution` | Not relevant to subspace content |
| `redrob_signals` | Structured data — belongs in Track B only |

### Token Limit Handling

INSTRUCTOR-large has a **512 token maximum input length** (inherited from T5-Large). A long career history description may exceed this after concatenation.

Mitigation strategy:
```python
MAX_TOKENS = 480  # Leave 32 tokens for instruction prefix

def truncate_career_text(text: str, tokenizer, max_tokens: int = MAX_TOKENS) -> str:
    tokens = tokenizer.encode(text)
    if len(tokens) <= max_tokens:
        return text
    # Truncate from the END — preserve most recent (most relevant) experience
    return tokenizer.decode(tokens[:max_tokens], skip_special_tokens=True)
```

Truncation from the end is the correct strategy because we sort history with most recent first — the most JD-relevant experience is at the beginning of the passage.

## 7. Batched Encoding — precompute.py

### The Correct Batching Pattern

```python
from InstructorEmbedding import INSTRUCTOR
import numpy as np
import torch

# Load model once
model = INSTRUCTOR('hkunlp/instructor-large')
model.eval()

# Build all career texts upfront
career_texts = [build_career_text(c) for c in candidates]  # list of 100K strings

# Build instruction-text pairs for each block
retrieval_pairs = [[RETRIEVAL_INSTRUCTION, t] for t in career_texts]
infra_pairs     = [[INFRA_INSTRUCTION, t] for t in career_texts]
eval_pairs      = [[EVAL_INSTRUCTION, t] for t in career_texts]

# Three batched passes — one per subspace
# batch_size=32 is safe for instructor-large on 6GB VRAM (RTX 4050)
with torch.no_grad():
    retrieval_vecs = model.encode(retrieval_pairs, batch_size=32, show_progress_bar=True)
    infra_vecs     = model.encode(infra_pairs,     batch_size=32, show_progress_bar=True)
    eval_vecs      = model.encode(eval_pairs,      batch_size=32, show_progress_bar=True)

# Concatenate into final 2304-d vectors
# Shape: [100K, 2304]
candidate_vecs = np.concatenate([retrieval_vecs, infra_vecs, eval_vecs], axis=1)
```

### Batch Size Selection for RTX 4050 (6GB VRAM)

| Batch Size | VRAM Usage | Speed |
|---|---|---|
| 8 | ~1.8GB | Safe but slow |
| 16 | ~2.4GB | Safe |
| **32** | **~3.2GB** | **Recommended — 2× headroom** |
| 64 | ~5.1GB | Risky — approaches 6GB limit |
| 128 | OOM | Do not use |

### INT8 Quantization (Optional Speedup)

The official INSTRUCTOR repository documents a 10× model size reduction via INT8 quantization with minimal quality loss:

```python
import torch
from InstructorEmbedding import INSTRUCTOR

model = INSTRUCTOR('hkunlp/instructor-large')

# Apply INT8 dynamic quantization to all Linear layers
model_int8 = torch.quantization.quantize_dynamic(
    model,
    {torch.nn.Linear},
    dtype=torch.qint8
)

# Use model_int8 for all encode calls
# Model size: ~1.3GB → ~130MB
# VRAM usage drops proportionally
# Allows larger batch sizes or leaves more headroom for FAISS index
```

**Quality impact:** Negligible for retrieval tasks. INT8 quantization preserves relative vector ordering, which is all that matters for FAISS similarity search.

## 8. JD Query Vector Construction

The JD query vector must be constructed using **identical instructions and identical model** as the candidate vectors. Any instruction mismatch means the query and candidate vectors live in different semantic spaces.

```python
# Extract JD text per subspace — take relevant sections from JD
JD_RETRIEVAL_TEXT = """
Production experience with embeddings-based retrieval systems deployed to real users.
Handling embedding drift, index refresh, retrieval-quality regression in production.
Production experience with vector databases or hybrid search infrastructure.
Shipped at least one end-to-end ranking or search or recommendation system.
"""

JD_INFRA_TEXT = """
Vector database scaling, FAISS, latency optimization, throughput, production deployment.
Background in distributed systems or large-scale inference optimization.
Deploy and maintain machine learning systems on cloud infrastructure.
"""

JD_EVAL_TEXT = """
Hands-on experience designing evaluation frameworks for ranking systems.
NDCG, MRR, MAP, offline-to-online correlation, A/B test interpretation.
Evaluation infrastructure, offline benchmarks, online A/B testing, recruiter feedback loops.
"""

# Encode JD query vectors
jd_retrieval = model.encode([[RETRIEVAL_INSTRUCTION, JD_RETRIEVAL_TEXT]])
jd_infra     = model.encode([[INFRA_INSTRUCTION, JD_INFRA_TEXT]])
jd_eval      = model.encode([[EVAL_INSTRUCTION, JD_EVAL_TEXT]])

# Concatenate into 2304-d query vector
jd_query_vec = np.concatenate([jd_retrieval, jd_infra, jd_eval], axis=1)
```

### Priority Weighting at Query Time

The JD weights retrieval experience most heavily. This is applied as a scalar multiplication **before** concatenation:

```python
RETRIEVAL_WEIGHT = 0.50
INFRA_WEIGHT     = 0.30
EVAL_WEIGHT      = 0.20

jd_query_vec = np.concatenate([
    jd_retrieval * RETRIEVAL_WEIGHT,
    jd_infra     * INFRA_WEIGHT,
    jd_eval      * EVAL_WEIGHT
], axis=1)
```

The same weights are applied to candidate vectors at index time, or applied only to the query vector at search time. Applying weights to the query at search time is more flexible — you can re-run searches with different weights without rebuilding the index.

## 9. FAISS HNSW Index — Updated Configuration

### Updated Index Parameters for 2304-d

```python
import faiss
import numpy as np

DIMENSION = 2304  # 3 × 768

# Build HNSW index with Inner Product metric
# M=32: number of neighbors per node (higher = better recall, more memory)
# efConstruction=200: construction-time search depth
index = faiss.IndexHNSWFlat(DIMENSION, 32, faiss.METRIC_INNER_PRODUCT)
index.hnsw.efSearch = 64  # search-time depth (higher = better recall, slower)

# Normalize candidate vectors before adding
# Required for Inner Product to behave like cosine (optional but recommended)
faiss.normalize_L2(candidate_vecs)

# Add all candidate vectors
index.add(candidate_vecs)

# Save index to disk
faiss.write_index(index, "track_a_hnsw.index")
```

### Why Not Cosine Similarity

FAISS `METRIC_INNER_PRODUCT` with L2-normalized vectors is mathematically equivalent to cosine similarity — but only if you normalize **per block independently** rather than globally.

If you normalize the full 2304-d vector globally, a candidate with a large retrieval block but near-zero eval block gets normalized upward overall, obscuring the weakness.

The correct approach for block-weighted scoring: normalize each 768-d block independently, then concatenate. This preserves intra-block magnitude relationships while making each block unit-normalized.

```python
def normalize_by_block(vecs: np.ndarray, block_size: int = 768) -> np.ndarray:
    n_blocks = vecs.shape[1] // block_size
    normalized_blocks = []
    for i in range(n_blocks):
        block = vecs[:, i*block_size:(i+1)*block_size].copy()
        faiss.normalize_L2(block)
        normalized_blocks.append(block)
    return np.concatenate(normalized_blocks, axis=1)

candidate_vecs_normalized = normalize_by_block(candidate_vecs)
index.add(candidate_vecs_normalized)
```

## 11. Updated rank.py Outline (Track A Steps Only)

```
rank.py — Track A steps
│
├── Step 1: Load FAISS index from disk → track_a_hnsw.index
│
├── Step 2: Load Track B Polars DataFrame from Parquet
│
├── Step 3: Build IDSelector bitmask from Track B hard filter
│          (years_exp BETWEEN 4 AND 12, is_honeypot == False)
│
├── Step 4: Construct JD query vector
│          - Encode JD_RETRIEVAL_TEXT, JD_INFRA_TEXT, JD_EVAL_TEXT
│          - Apply priority weights (0.5, 0.3, 0.2)
│          - Concatenate → [1 × 2304]
│          - Per-block L2 normalize
│
├── Step 5: FAISS HNSW search with IDSelector bitmask
│          index.search(jd_query_vec, k=300) → Top 300 candidate IDs + scores
│
├── Step 6: BM25 sparse search (Track A hybrid component)
│          Combine with FAISS results via RRF → Top 300
│
└── Step 7: Pass Top 300 candidate IDs to Track B LightGBM reranker
```

**Note:** INSTRUCTOR model is NOT loaded during rank.py. The JD query vector is either pre-computed and saved to disk during precompute.py, or the model is loaded once at rank.py startup for the JD encoding only (4 encode calls — one per subspace + one overall for Track B feature). This is negligible compute.

The hackathon constraint says NO `import torch` during online execution (rank.py crashes RAM). This constraint applies to rank.py only.

INSTRUCTOR is used **only in precompute.py** (offline). All outputs are saved as numpy arrays and FAISS index files. rank.py loads the pre-built FAISS index and never imports torch or InstructorEmbedding.
